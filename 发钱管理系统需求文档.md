# 发钱管理系统 - 需求文档

## 一、项目概述

### 1.1 项目背景
开发一个简单的发钱管理系统，用于管理发钱记录、收款人信息、账户余额等。系统需要支持基本的增删改查操作，并提供统计分析功能。

### 1.2 技术栈
- **语言**: Java
- **数据库**: MySQL 8.0+
- **数据库访问**: JDBC + PreparedStatement（**不使用 Spring/MyBatis**）
  - **JDBC**: Java 连接数据库的标准接口（类似"翻译官"）
  - **PreparedStatement**: 预编译 SQL 语句，防止 SQL 注入攻击，性能更好
- **项目结构**: 渐进式学习结构（从简单到复杂，每个模块可独立测试）

### 1.3 为什么不用 Spring？

**核心原因：先学"手动挡"，再学"自动挡"**

| 对比项 | 原生 JDBC（本项目） | Spring + MyBatis（以后） |
|--------|-------------------|------------------------|
| 学习难度 | 需要理解底层原理 | 上手快，但不懂原理 |
| 代码量 | 多（手动管理连接、事务） | 少（自动化处理） |
| 理解深度 | 清楚每一步在做什么 | 容易"知其然不知所以为" |
| 调试能力 | 出错能快速定位 | 出错不知道从哪查 |
| 面试价值 | JDBC 是必考基础 | 会用但说不清原理会减分 |

**学习路径：**
```
第一阶段（现在）：原生 JDBC           ← 你在这里
├─ 手动管理数据库连接
├─ 手写 SQL 语句
├─ 手动处理事务（commit/rollback）
├─ 理解 PreparedStatement 防注入原理
└─ 掌握 ResultSet 结果集处理

第二阶段（3-6个月后）：Spring Boot + MyBatis
├─ 依赖注入（@Autowired）
├─ 声明式事务（@Transactional）
├─ MyBatis XML/注解映射
└─ 这时你会理解：Spring 帮你自动做了什么

第三阶段（1年后）：Spring Cloud 微服务
```

**比喻：**
- 原生 JDBC = 手动挡汽车（学会了才知道离合、档位的作用）
- Spring = 自动挡汽车（开起来很爽，但不懂原理）

**完成本项目后的收获：**
- ✅ 面试被问"JDBC 怎么用"不会懵
- ✅ 以后学 Spring 时会恍然大悟
- ✅ 出 Bug 能快速定位到底层原因
- ✅ 真正理解事务、连接池、SQL 注入等概念

### 1.4 学习目标
- 掌握 CRUD 操作（增删改查）
- 理解数据库设计（主键、外键、索引）
- 熟练使用 JDBC 和 PreparedStatement
- 掌握事务处理
- 学习统计查询（GROUP BY、JOIN、聚合函数）

## 二、功能需求

### 2.1 用户管理模块

#### 功能描述
管理收款人信息，包括姓名、账号、联系方式等。

#### 功能点
1. **添加用户**
   - 输入：姓名、手机号、银行卡号、开户行
   - 校验：手机号格式、银行卡号唯一性
   - 输出：成功/失败信息

2. **查询用户**
   - 按用户ID查询
   - 按姓名模糊查询
   - 按手机号查询
   - 查询所有用户列表

3. **更新用户信息**
   - 可修改：姓名、手机号、开户行
   - 不可修改：用户ID、银行卡号

4. **删除用户**
   - 删除前检查：该用户是否有未完成的发钱记录
   - 软删除：标记为已删除，不物理删除

---

### 2.2 账户管理模块

#### 功能描述
管理发钱方的账户余额，支持充值、余额查询等。

#### 功能点
1. **创建账户**
   - 输入：账户名称、初始余额
   - 默认状态：激活

2. **充值**
   - 输入：账户ID、充值金额
   - 校验：金额必须大于0
   - 记录充值流水

3. **查询余额**
   - 按账户ID查询当前余额
   - 查询所有账户列表

4. **账户冻结/解冻**
   - 冻结后不能发钱
   - 可以随时解冻

---

### 2.3 发钱记录模块

#### 功能描述
记录每一笔发钱操作，包括发钱人、收钱人、金额、时间、状态等。

#### 功能点
1. **创建发钱记录**
   - 输入：发钱账户ID、收款用户ID、金额、备注
   - 校验：
     - 账户余额是否充足
     - 账户是否冻结
     - 收款用户是否存在
     - 金额必须大于0
   - 事务处理：
     - 扣减发钱账户余额
     - 创建发钱记录
     - 两步操作必须同时成功或失败

2. **查询发钱记录**
   - 按记录ID查询
   - 按发钱账户ID查询
   - 按收款用户ID查询
   - 按时间范围查询
   - 按状态查询（待处理/已完成/已取消）
   - 分页查询

3. **更新发钱状态**
   - 待处理 → 已完成
   - 待处理 → 已取消（退款，恢复余额）

4. **删除发钱记录**
   - 只能删除已取消的记录
   - 软删除

---

### 2.4 统计报表模块

#### 功能描述
提供各种统计分析功能，帮助了解发钱情况。

#### 功能点
1. **发钱统计**
   - 统计每个账户的总发钱金额
   - 统计每个账户的发钱次数
   - 统计时间段内的发钱总额

2. **收款统计**
   - 统计每个用户的总收款金额
   - 统计每个用户的收款次数
   - 排行榜：收款金额最多的用户

3. **流水统计**
   - 按天统计发钱金额
   - 按月统计发钱金额
   - 按状态统计发钱记录数

4. **余额预警**
   - 查询余额低于阈值的账户
   - 余额不足提醒

## 三、数据库设计

### 3.1 用户表（users）

| 字段名 | 数据类型 | 约束 | 说明 |
|--------|---------|------|------|
| id | INT | PRIMARY KEY, AUTO_INCREMENT | 用户ID |
| name | VARCHAR(50) | NOT NULL | 用户姓名 |
| phone | VARCHAR(20) | NOT NULL, UNIQUE | 手机号 |
| bank_card | VARCHAR(30) | NOT NULL, UNIQUE | 银行卡号 |
| bank_name | VARCHAR(100) | NOT NULL | 开户行 |
| status | VARCHAR(20) | NOT NULL, DEFAULT 'ACTIVE' | 状态：ACTIVE/DELETED |
| create_time | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| update_time | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | 更新时间 |

**索引：**
- PRIMARY KEY: id
- UNIQUE INDEX: phone, bank_card
- INDEX: name（用于模糊查询）

---

### 3.2 账户表（accounts）

| 字段名 | 数据类型 | 约束 | 说明 |
|--------|---------|------|------|
| id | INT | PRIMARY KEY, AUTO_INCREMENT | 账户ID |
| name | VARCHAR(100) | NOT NULL | 账户名称 |
| balance | DECIMAL(15,2) | NOT NULL, DEFAULT 0.00 | 当前余额 |
| status | VARCHAR(20) | NOT NULL, DEFAULT 'ACTIVE' | 状态：ACTIVE/FROZEN |
| create_time | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| update_time | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | 更新时间 |

**索引：**
- PRIMARY KEY: id

**约束：**
- CHECK: balance >= 0（余额不能为负）

---

### 3.3 发钱记录表（payment_records）

| 字段名 | 数据类型 | 约束 | 说明 |
|--------|---------|------|------|
| id | INT | PRIMARY KEY, AUTO_INCREMENT | 记录ID |
| account_id | INT | NOT NULL, FOREIGN KEY | 发钱账户ID |
| user_id | INT | NOT NULL, FOREIGN KEY | 收款用户ID |
| amount | DECIMAL(15,2) | NOT NULL | 发钱金额 |
| status | VARCHAR(20) | NOT NULL, DEFAULT 'PENDING' | 状态：PENDING/COMPLETED/CANCELLED |
| remark | VARCHAR(500) | | 备注 |
| create_time | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| update_time | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | 更新时间 |

**索引：**
- PRIMARY KEY: id
- INDEX: account_id（查询某账户的发钱记录）
- INDEX: user_id（查询某用户的收款记录）
- INDEX: create_time（按时间查询）
- INDEX: status（按状态查询）

**外键：**
- account_id REFERENCES accounts(id)
- user_id REFERENCES users(id)

**约束：**
- CHECK: amount > 0（金额必须大于0）

---

### 3.4 充值记录表（recharge_records）

| 字段名 | 数据类型 | 约束 | 说明 |
|--------|---------|------|------|
| id | INT | PRIMARY KEY, AUTO_INCREMENT | 记录ID |
| account_id | INT | NOT NULL, FOREIGN KEY | 账户ID |
| amount | DECIMAL(15,2) | NOT NULL | 充值金额 |
| before_balance | DECIMAL(15,2) | NOT NULL | 充值前余额 |
| after_balance | DECIMAL(15,2) | NOT NULL | 充值后余额 |
| create_time | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | 充值时间 |

**索引：**
- PRIMARY KEY: id
- INDEX: account_id（查询某账户的充值记录）

**外键：**
- account_id REFERENCES accounts(id)

**约束：**
- CHECK: amount > 0

---

### 3.5 ER 图关系

```
    ↓ N:1
```

## 四、核心业务逻辑

### 4.1 发钱流程（事务处理）

```
开始事务
    1. 检查发钱账户状态（是否冻结）
    2. 检查账户余额是否充足
    3. 检查收款用户是否存在且有效
    4. 扣减发钱账户余额
        UPDATE accounts SET balance = balance - ? WHERE id = ?
    5. 创建发钱记录
        INSERT INTO payment_records (...) VALUES (...)
    6. 提交事务
异常处理
    回滚事务
    返回失败信息
```

**关键点：**
- 使用数据库事务保证原子性
- 余额检查和扣款必须在同一个事务中
- 防止并发问题（可使用悲观锁：SELECT ... FOR UPDATE）

---

### 4.2 取消发钱流程（退款）

```
开始事务
    1. 检查发钱记录状态（只能取消待处理状态）
    2. 恢复发钱账户余额
        UPDATE accounts SET balance = balance + ? WHERE id = ?
    3. 更新发钱记录状态为已取消
        UPDATE payment_records SET status = 'CANCELLED' WHERE id = ?
    4. 提交事务
异常处理
    回滚事务
```

---

### 4.3 充值流程

```
开始事务
    1. 查询当前余额
        SELECT balance FROM accounts WHERE id = ? FOR UPDATE
    2. 更新余额
        UPDATE accounts SET balance = balance + ? WHERE id = ?
    3. 记录充值流水
        INSERT INTO recharge_records (...) VALUES (...)
    4. 提交事务
异常处理
    回滚事务
```

## 五、非功能需求

### 5.1 性能要求
- 单次查询响应时间 < 100ms
- 支持并发操作（使用事务隔离）
- 数据量：预计用户1000+，发钱记录10000+

### 5.2 安全要求
- 所有数据库操作使用 PreparedStatement（防止SQL注入）
- 敏感信息（银行卡号）脱敏显示
- 关键操作（发钱、充值）记录日志

### 5.3 数据完整性
- 使用外键约束保证数据一致性
- 使用事务保证操作原子性
- 余额不能为负数（CHECK约束）

### 5.4 可扩展性
- 代码分层：DAO层、Service层、Main层
- 使用接口，便于后续扩展（如切换数据库）

### 6.2 渐进式开发流程

**核心思想**：每个 Step 文件都能独立运行，逐步验证功能

| Step | 文件名 | 学习重点 | 可以独立运行？ |
|------|--------|---------|--------------|
| 1 | Step1_DatabaseSetup.java | 创建表、理解表结构 | ✅ 是 |
| 2 | Step2_EntityTest.java | 实体类、getter/setter | ✅ 是 |
| 3 | Step3_UserDaoTest.java | JDBC、PreparedStatement | ✅ 是 |
| 4 | Step4_AccountDaoTest.java | DAO 设计模式 | ✅ 是 |
| 5 | Step5_PaymentTest.java | 事务处理、回滚 | ✅ 是 |
| 6 | Step6_StatisticsTest.java | 复杂查询、JOIN | ✅ 是 |
| 7 | Main.java | 整合所有功能 | ✅ 是 |

**示例：Step3_UserDaoTest.java**
```java

```

**运行方式**：
```bash
# 方式1：命令行运行
cd src
javac moneyTransfer/Step3_UserDaoTest.java
java moneyTransfer.Step3_UserDaoTest

# 方式2：IDE 中右键运行
```

### 6.4 为什么这样设计？

**优点：**
1. **逐步学习**：从简单到复杂，不会一上来就懵
2. **快速验证**：写完一个模块立即测试，不用等全部写完
3. **方便调试**：哪个模块有问题，直接运行对应的 Step 文件
4. **独立性强**：Step 文件之间互不影响，可以反复运行
5. **学习曲线友好**：每个 Step 只关注一个知识点

**对比传统结构：**
```
❌ 传统方式：写完所有代码 → 运行 Main → 出错了不知道哪里错
✅ 渐进方式：写一个模块 → 运行 Step → 测试通过 → 继续下一个
```

## 九、SQL 语句示例

### 9.1 查询某账户的发钱记录（带用户信息）
```sql
SELECT 
    pr.id AS 记录ID,
    pr.amount AS 金额,
    u.name AS 收款人,
    u.phone AS 手机号,
    pr.status AS 状态,
    pr.create_time AS 创建时间
FROM payment_records pr
INNER JOIN users u ON pr.user_id = u.id
WHERE pr.account_id = ?
ORDER BY pr.create_time DESC
LIMIT 10;
```

### 9.2 统计每个账户的发钱总额
```sql
SELECT 
    a.name AS 账户名称,
    COUNT(pr.id) AS 发钱次数,
    SUM(pr.amount) AS 总发钱金额,
    a.balance AS 当前余额
FROM accounts a
LEFT JOIN payment_records pr ON a.id = pr.account_id 
    AND pr.status = 'COMPLETED'
GROUP BY a.id, a.name, a.balance
ORDER BY SUM(pr.amount) DESC;
```

### 9.3 查询收款金额前10的用户
```sql
SELECT 
    u.name AS 用户姓名,
    COUNT(pr.id) AS 收款次数,
    SUM(pr.amount) AS 总收款金额
FROM users u
INNER JOIN payment_records pr ON u.id = pr.user_id
WHERE pr.status = 'COMPLETED'
GROUP BY u.id, u.name
ORDER BY SUM(pr.amount) DESC
LIMIT 10;
```

### 9.4 按天统计发钱金额
```sql
SELECT 
    DATE(create_time) AS 日期,
    COUNT(id) AS 发钱次数,
    SUM(amount) AS 总金额
FROM payment_records
WHERE status = 'COMPLETED'
    AND create_time >= ?
    AND create_time < ?
GROUP BY DATE(create_time)
ORDER BY DATE(create_time);
```

### 9.5 查询余额低于阈值的账户
```sql
SELECT 
    id AS 账户ID,
    name AS 账户名称,
    balance AS 当前余额
FROM accounts
WHERE balance < ?
    AND status = 'ACTIVE'
ORDER BY balance ASC;
```

## 十、关键技术点

### 10.1 JDBC 连接管理

**什么是 JDBC？**
- Java Database Connectivity（Java 数据库连接）
- Java 操作数据库的标准 API
- 就像"翻译官"，让 Java 能和 MySQL 对话

**连接步骤：**
```java
// 1. 加载驱动（MySQL 8.0 会自动加载）
Class.forName("com.mysql.cj.jdbc.Driver");

// 2. 建立连接
String url = "jdbc:mysql://localhost:3306/money_transfer_db?useSSL=false&serverTimezone=UTC";
String user = "root";
String password = "your_password";
Connection conn = DriverManager.getConnection(url, user, password);

// 3. 使用连接执行 SQL
// ...

// 4. 关闭连接（重要！）
conn.close();
```

**最佳实践：**
- 使用 try-with-resources 自动关闭资源
- 避免连接泄漏
- 生产环境使用连接池（HikariCP、Druid）

### 10.2 PreparedStatement 完整使用示例

**为什么用 PreparedStatement？**
```java
// ❌ 不推荐：Statement（容易被 SQL 注入攻击）
String name = "'; DROP TABLE users; --";  // 恶意输入
String sql = "SELECT * FROM users WHERE name = '" + name + "'";
// 实际执行：SELECT * FROM users WHERE name = ''; DROP TABLE users; --'
// 你的表被删了！

// ✅ 推荐：PreparedStatement（自动转义，安全）
String sql = "SELECT * FROM users WHERE name = ?";
PreparedStatement pstmt = conn.prepareStatement(sql);
pstmt.setString(1, name);  // 自动转义，name 被当成普通字符串
// 实际执行：SELECT * FROM users WHERE name = '\'; DROP TABLE users; --'
// 安全！
```

**INSERT 示例：**
```java
String sql = "INSERT INTO users (name, phone, bank_card, bank_name) VALUES (?, ?, ?, ?)";
try (PreparedStatement pstmt = conn.prepareStatement(sql)) {
    pstmt.setString(1, "张三");           // 第1个 ? 替换为 "张三"
    pstmt.setString(2, "13800138000");   // 第2个 ?
    pstmt.setString(3, "6222000012345"); // 第3个 ?
    pstmt.setString(4, "工商银行");       // 第4个 ?
    
    int rows = pstmt.executeUpdate();    // 返回影响的行数
    System.out.println("插入成功，影响行数: " + rows);
}
```

**SELECT 示例：**
```java
String sql = "SELECT id, name, phone FROM users WHERE phone = ?";
try (PreparedStatement pstmt = conn.prepareStatement(sql)) {
    pstmt.setString(1, "13800138000");
    
    ResultSet rs = pstmt.executeQuery();  // 执行查询，返回结果集
    while (rs.next()) {
        int id = rs.getInt("id");         // 根据列名获取值
        String name = rs.getString("name");
        String phone = rs.getString("phone");
        System.out.println(id + " - " + name + " - " + phone);
    }
    rs.close();
}
```

**UPDATE 示例：**
```java
String sql = "UPDATE users SET phone = ? WHERE id = ?";
try (PreparedStatement pstmt = conn.prepareStatement(sql)) {
    pstmt.setString(1, "13900139000");  // 新手机号
    pstmt.setInt(2, 1);                 // 用户ID
    
    int rows = pstmt.executeUpdate();
    System.out.println("更新成功，影响行数: " + rows);
}
```

**DELETE 示例：**
```java
String sql = "DELETE FROM users WHERE id = ?";
try (PreparedStatement pstmt = conn.prepareStatement(sql)) {
    pstmt.setInt(1, 1);
    
    int rows = pstmt.executeUpdate();
    System.out.println("删除成功，影响行数: " + rows);
}
```

**常用方法：**
```java
// 设置参数（索引从 1 开始）
pstmt.setString(1, "字符串");
pstmt.setInt(2, 100);
pstmt.setDouble(3, 99.99);
pstmt.setTimestamp(4, new Timestamp(System.currentTimeMillis()));

// 执行 SQL
int rows = pstmt.executeUpdate();     // INSERT/UPDATE/DELETE，返回影响行数
ResultSet rs = pstmt.executeQuery();  // SELECT，返回结果集

// 获取结果集数据
rs.getInt("id")           // 根据列名
rs.getString(2)           // 根据列索引（从1开始）
rs.getDouble("balance")
rs.getTimestamp("create_time")
```

### 10.3 事务处理
```java
Connection conn = null;
try {
    conn = DBUtil.getConnection();
    conn.setAutoCommit(false);  // 开启事务
    
    // 执行多个SQL操作
    // ...
    
    conn.commit();  // 提交事务
} catch (Exception e) {
    if (conn != null) {
        conn.rollback();  // 回滚事务
    }
    throw e;
} finally {
    if (conn != null) {
        conn.setAutoCommit(true);
        conn.close();
    }
}
```

### 10.4 并发控制（悲观锁）
```sql
-- 查询时加锁，防止并发问题
SELECT balance FROM accounts WHERE id = ? FOR UPDATE;
```

## 附录：常见问题

### Q1: 为什么选择 MySQL？
A: MySQL 是工业界最常用的关系型数据库，学会了可以直接用于实际项目。而且你可以直观看到数据的变化，方便学习和调试。

### Q2: 什么时候需要使用事务？
A: 当多个 SQL 操作需要保证原子性时（要么全成功，要么全失败），就需要使用事务。例如：发钱操作需要同时扣款和创建记录。

### Q3: PreparedStatement 和 Statement 有什么区别？
A: PreparedStatement 预编译 SQL，性能更好，且能防止 SQL 注入攻击。应该优先使用 PreparedStatement。

### Q4: 如何防止并发导致的余额超发？
A: 使用 `SELECT ... FOR UPDATE` 悲观锁，或者在数据库层面添加 CHECK 约束（balance >= 0）。

### Q5: 软删除和硬删除有什么区别？
A: 软删除只标记删除状态，数据仍在数据库中，可以恢复。硬删除直接从数据库删除数据，不可恢复。一般推荐使用软删除。
